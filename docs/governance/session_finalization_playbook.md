# Session Finalization Playbook

**Статус:** Canonical  
**Версия:** v1.0

## Назначение

Настоящий playbook описывает **операционный алгоритм корректного закрытия сессии**
в соответствии с требованиями `session_closure_protocol.md`.

Документ:
- не вводит новых правил;
- не переопределяет обязательства Protocol;
- описывает пошаговый порядок действий и контрольные проверки.

---

## 1. Область применения

Playbook применяется:
- при любом закрытии рабочей сессии;
- при переходе в новый чат;
- при превентивном завершении сессии из-за риска потери контекста.

---

## 2. Общие принципы выполнения

1. Все действия выполняются **последовательно**.
2. Каждый шаг должен быть **проверяемым**.
3. Отсутствие обязательного шага считается ошибкой закрытия.
4. Любые сомнения трактуются **в пользу дополнительной фиксации**.

---

## 3. Пошаговый алгоритм закрытия

### Шаг 1. Определение необходимости закрытия

Проверить условия из `session_closure_protocol.md, §2`.

Если хотя бы одно условие выполнено — закрытие **обязательно**.

---

### Шаг 2. Сбор фактов сессии

Зафиксировать:
- цель сессии;
- фактически выполненные действия;
- изменения (code / docs / server);
- принятые решения;
- нерешённые вопросы.

---

### Шаг 3. Подготовка Session Closure Log

Создать файл:

`session_closure_log_YYYY_MM_DD_vX.Y.md`

Обязательные разделы — согласно  
`documentation_rules_style_guide.md, §4.2`.

---

### Шаг 4. Проверка необходимости обновления других артефактов

Проверить, требуется ли обновление:
- `project_summary_vX.Y.md`
- `project_history_log.md`
- `techspec_vX.Y.md`
- `plan_vX.Y.md`

Если обновление требуется:
- внести изменения;
- увеличить версию;
- отразить изменения в closure log.

---

### Шаг 5. Внутренняя согласованность

Проверить:
- отсутствие противоречий между артефактами;
- согласованность версий;
- корректность ссылок на документы.

---

### Шаг 6. Финальная проверка

Закрытие считается корректным, если:
- все обязательные артефакты созданы или обновлены;
- версии увеличены при необходимости;
- closure log содержит проверяемые факты.

---

## 4. Decision Matrix

| Условие | Действие |
|-------|---------|
| Были приняты решения | Closure log обязателен |
| Были изменения | Closure log + обновление summary |
| Обновлён TechSpec | Увеличить версию TechSpec |
| Нет изменений | Допускается краткий closure log |
| Риск потери контекста | Немедленное закрытие с фиксацией |

---

## 5. Типовые ошибки

1. Отсутствие closure log.
2. Обновление документов без версии.
3. Противоречия между summary и log.
4. Использование контекста чата вместо артефактов.
