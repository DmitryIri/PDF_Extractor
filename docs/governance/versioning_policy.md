# Versioning Policy

**Статус:** Canonical

**Версия:** v1.0

**Назначение:**
Данный документ фиксирует **канонические правила версионирования** для всех артефактов проекта, используемых в совместной работе пользователя и AI.

Цели:
- детерминированность и воспроизводимость изменений;
- аудитопригодность;
- запрет «тихих» ретроспективных правок;
- единая семантика Major/Minor для всех канонических документов.

---

## 1. Общие правила

1. **Единый формат версии:** `v<MAJOR>.<MINOR>` (SemVer для документации).
2. **Нулевая версия запрещена** для канонических артефактов. Первый канонический релиз — `v1.0`.
3. **Ретроспективные правки запрещены.** Любая правка оформляется выпуском новой версии.
4. **Один документ — одна версия.** Версия указана в заголовке документа и (если принято) в имени файла.
5. **Изменения должны быть объяснимы**: каждая версия содержит `CHANGELOG`.

---

## 2. Семантика Major и Minor

### 2.1 MAJOR (vN+1.0)
MAJOR повышается, если выполняется хотя бы одно условие:
- изменены **требования** (scope, обязательные инварианты, критерии корректности результата);
- изменена **архитектура** (границы компонентов, ответственность сервисов, интерфейсы/контракты);
- изменены **правила процесса**, влияющие на обязательные артефакты или порядок работ (bootstrap/closure);
- введён новый канонический артефакт (добавлен в `canonical_artifact_registry.md`);
- изменены правила именования, влияющие на воспроизводимость.

### 2.2 MINOR (vN.M+1)
MINOR повышается, если:
- уточнены формулировки без изменения смысла требований/архитектуры;
- добавлены примеры, чек-листы, инструкции выполнения;
- зафиксирован прогресс работ, статус шагов, handoff между чатами;
- исправлены ошибки/неточности без изменения обязательств.

---

## 3. Политика версий по типам артефактов

### 3.1 TechSpec (`techspec_vX.Y.md`)
- MAJOR: изменены требования/архитектура/контракты.
- MINOR: уточнения текста, примеры, расширение пояснений без изменения требований.

### 3.2 Plan (`plan_vX.Y.md`)
- MAJOR: изменена структура плана/этапов, добавлены/удалены крупные фазы, изменён порядок фаз.
- MINOR: обновлён статус шагов, добавлены подпункты, уточнены критерии завершения.

### 3.3 Project Summary (`project_summary_vX.Y.md`)
- MAJOR: изменилось состояние проекта из-за крупных архитектурных/требовательных изменений.
- MINOR: регулярное обновление статуса, handoff, фиксация прогресса без смены курса.

### 3.4 Session Closure Log (`session_closure_log_YYYY_MM_DD_vX.Y.md`)
- MAJOR: изменена форма/структура лога как стандарта (редко; обычно это изменение шаблона, а не конкретного лога).
- MINOR: исправления/дополнения конкретного лога при выявлении пропусков (при этом сохранять детерминизм и ссылаться на причину исправления).

### 3.5 Project History Log (`project_history_log.md`)
- Версия файла может не инкрементироваться в имени (файл кумулятивный), но **обязан иметь внутренний `CHANGELOG`** с отметками версий/дат.
- Правило: записи добавляются только в конец, ретро-редактирование запрещено.

### 3.6 Meta-документы (Registry / Bootstrap / Playbook / Style)
- MAJOR: изменены обязательные правила взаимодействия.
- MINOR: уточнения, примеры, устранение неоднозначностей.

---

## 4. Правила именования (канон)

Канонические имена файлов:
- `project_summary_vX.Y.md`
- `session_closure_log_YYYY_MM_DD_vX.Y.md`
- `project_history_log.md`
- `techspec_vX.Y.md`
- `plan_vX.Y.md`

Примечания:
- `YYYY_MM_DD` — дата по локальному времени пользователя.
- Допускается хранение архивных версий (старые `vX.Y`) без перезаписи.

---

## 5. CHANGELOG (обязателен)

Каждая версия документа обязана иметь раздел `CHANGELOG`.
Формат записи:
- `vX.Y — YYYY-MM-DD — кратко: что изменено и почему`.

---

## 6. Нормативное правило

Если документ нарушает правила версионирования, он **не может считаться каноническим** до исправления (выпуском новой версии).
