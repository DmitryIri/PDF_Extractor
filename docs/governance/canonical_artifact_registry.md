# Canonical Artifact Registry

**Статус:** Canonical  
**Версия:** v1.0

## Назначение

Данный документ определяет **канонический перечень допустимых артефактов проекта**, их типы, роли и обязательность.

Реестр служит:
- единым источником истины по документам проекта;
- нормативной основой для bootstrap-процедур и закрытия сессий;
- механизмом предотвращения неявных, устных и контекстно-зависимых договорённостей.

Любой документ, **не перечисленный в настоящем реестре**, не считается каноническим и не должен использоваться в работе без явного расширения реестра.

---

## 1. Принципы реестра

1. **Явность** — каждый артефакт имеет зафиксированную роль и назначение.
2. **Типизация** — каждый артефакт относится к строго определённому типу.
3. **Обязательность** — для каждого артефакта указано, является ли он обязательным.
4. **Единый центр истины** — данный документ имеет приоритет над любыми упоминаниями артефактов в других файлах.
5. **Расширяемость** — добавление нового типа артефакта допускается только через обновление настоящего реестра с увеличением версии.

---

## 2. Meta-артефакты (Governance)

### 2.1 Canonical Artifact Registry
- **Тип:** Meta
- **Обязательность:** всегда
- **Роль:**
  - определяет допустимый набор артефактов проекта;
  - является единым источником истины по типам и ролям документов.

---

### 2.2 Context Bootstrap Protocol
- **Тип:** Meta
- **Обязательность:** всегда
- **Роль:**
  - определяет порядок инициализации контекста новой сессии;
  - фиксирует минимальный и расширенный bootstrap-пакет документов;
  - задаёт порядок интерпретации артефактов.

---

### 2.3 Session Closure Protocol
- **Тип:** Meta
- **Обязательность:** всегда
- **Роль:**
  - определяет минимальные условия корректного закрытия сессии;
  - выступает нормативным «законом» завершения работы;
  - не содержит операционных инструкций.

---

### 2.4 Session Finalization Playbook
- **Тип:** Meta
- **Обязательность:** всегда
- **Роль:**
  - описывает операционный алгоритм закрытия сессии;
  - реализует требования Session Closure Protocol;
  - содержит decision-matrix и чек-листы.

---

### 2.5 Versioning Policy
- **Тип:** Meta
- **Обязательность:** всегда
- **Роль:**
  - определяет правила Major/Minor-версионирования;
  - запрещает ретроспективные правки;
  - обеспечивает аудитопригодность изменений.

---

### 2.6 Documentation Rules & Style Guide
- **Тип:** Meta
- **Обязательность:** всегда
- **Роль:**
  - определяет структуру, стиль и правила оформления документов;
  - обеспечивает терминологическую и структурную согласованность;
  - задаёт требования к ссылкам и формулировкам.

---

## 3. State-артефакты (As-Is)

### 3.1 Project Summary
- **Файл:** `project_summary_vX.Y.md`
- **Тип:** State
- **Обязательность:** всегда
- **Роль:**
  - фиксирует текущее состояние проекта;
  - служит основной точкой восстановления контекста.

---

### 3.2 Session Closure Log
- **Файл:** `session_closure_log_YYYY_MM_DD_vX.Y.md`
- **Тип:** State
- **Обязательность:** всегда при закрытии сессии
- **Роль:**
  - фиксирует фактически выполненную работу за сессию;
  - обеспечивает воспроизводимый handoff между сессиями.

---

### 3.3 Project History Log
- **Файл:** `project_history_log.md`
- **Тип:** State
- **Обязательность:** всегда
- **Роль:**
  - ведёт кумулятивную историю проекта;
  - отражает ключевые этапы и вехи.

---

## 4. Design-артефакты (To-Be)

### 4.1 Technical Specification (TechSpec)
- **Файл:** `techspec_vX.Y.md`
- **Тип:** Design
- **Обязательность:** при наличии проектирования
- **Роль:**
  - фиксирует целевую архитектуру и требования;
  - определяет инварианты системы.

---

### 4.2 Plan / Roadmap
- **Файл:** `plan_vX.Y.md`
- **Тип:** Design
- **Обязательность:** при поэтапной реализации
- **Роль:**
  - определяет этапы и шаги реализации;
  - задаёт критерии завершения.

---

## 5. Правило приоритета

При конфликте между документами приоритет применяется в следующем порядке:

1. Canonical Artifact Registry
2. Governance-артефакты
3. State-артефакты
4. Design-артефакты
