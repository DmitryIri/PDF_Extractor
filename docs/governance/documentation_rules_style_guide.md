# Documentation Rules & Style Guide

**Статус:** Canonical

**Версия:** v1.0

**Назначение:**
Данный документ фиксирует **обязательные правила оформления, структуры и стиля** всех канонических артефактов проекта.

Цель документа — обеспечить:
- детерминированность и воспроизводимость документации;
- единый стиль и структуру;
- исключение неявных состояний и устных договорённостей;
- возможность восстановления контекста третьим лицом без участия авторов.

---

## 1. Общие принципы

1. **Facts-only**: в документации фиксируются только проверяемые факты, решения и состояния.
2. **No implicit knowledge**: запрещены формулировки вида «обсуждали», «очевидно», «как договорились ранее» без ссылки на артефакт.
3. **Reproducibility-first**: любой инженер должен иметь возможность продолжить работу, опираясь только на документы.
4. **Documentation = system**: документация является частью системы и подчиняется тем же правилам качества, что и код.

---

## 2. Общие правила оформления

1. Формат документов — `Markdown (.md)`.
2. Заголовок первого уровня (`#`) — название документа.
3. В начале документа **обязательно** указываются:
   - Статус (Canonical / Draft / Deprecated);
   - Версия (`vX.Y`);
   - Назначение (1–2 абзаца).
4. Каждый документ **обязан** содержать раздел `CHANGELOG`.
5. Изменения без увеличения версии запрещены.

---

## 3. Правила ссылок (канон)

1. В документации **не используются URL** для ссылок между артефактами.
2. Ссылка оформляется строго в виде:

   `<filename> vX.Y, §N (название раздела)`

3. Ссылка **обязательна** при:
   - упоминании решений;
   - упоминании требований;
   - ссылке на правила или протоколы.

Примеры:
- `canonical_artifact_registry.md v1.0, §8 (Сводная таблица)`
- `session_closure_protocol.md v1.0, §5 (Обязательные артефакты)`

---

## 4. Каноническая структура артефактов

### 4.1 Project Summary (`project_summary_vX.Y.md`)

**Обязательные разделы:**
1. Назначение
2. Текущее состояние проекта
3. Архитектура (as-is)
4. Статус компонентов
5. Принятые решения (с ссылками)
6. Открытые вопросы / риски
7. Следующий шаг
8. CHANGELOG

---

### 4.2 Session Closure Log (`session_closure_log_YYYY_MM_DD_vX.Y.md`)

**Обязательные разделы:**
1. Meta (дата, версия, scope сессии)
2. Цель сессии
3. Что было сделано (пошагово)
4. Изменения (code / docs / server)
5. Принятые решения
6. Риски / проблемы
7. Открытые вопросы
8. Точка остановки
9. Ссылки на актуальные документы
10. CHANGELOG

Каждый шаг обязан быть проверяемым (команда → результат → факт).

---

### 4.3 Project History Log (`project_history_log.md`)

**Обязательные правила:**
- файл кумулятивный;
- записи добавляются только в конец;
- каждая запись содержит дату, краткое описание, ссылки на версии документов;
- ретроспективное редактирование запрещено.

---

### 4.4 Technical Specification (`techspec_vX.Y.md`)

**Обязательные разделы:**
1. Назначение
2. Область применения
3. Инварианты и ограничения
4. Архитектура (to-be)
5. Контракты / интерфейсы
6. Нефункциональные требования
7. CHANGELOG

---

### 4.5 Plan / Roadmap (`plan_vX.Y.md`)

**Обязательные разделы:**
1. Назначение
2. Этапы и шаги
3. Критерии завершения этапов
4. Текущий статус
5. Риски и зависимости
6. CHANGELOG

---

## 5. Язык и стиль

1. Стиль — технический, нейтральный.
2. Запрещены эмоциональные формулировки и оценочные суждения.
3. Предпочтение коротким, однозначным предложениям.
4. Термины используются последовательно и без синонимической замены.

---

## 6. Запреты (жёстко)

Запрещено:
- устные договорённости без фиксации;
- ссылки на контекст чата вместо документов;
- «тихие» правки без версии;
- удаление исторических записей;
- использование неоднозначных формулировок.

---

## 7. Контроль качества документации

Документ считается некорректным, если:
- отсутствует версия или changelog;
- нарушена структура;
- есть неявные ссылки;
- невозможно воспроизвести состояние проекта.

---

## 8. Нормативное правило

Документы, не соответствующие настоящему Style Guide, **не считаются каноническими** до исправления.
