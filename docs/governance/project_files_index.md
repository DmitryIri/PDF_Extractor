# PROJECT_FILES_INDEX

**Проект:** PDF Extractor  
**Статус:** Canonical  
**Назначение:** Единая точка навигации по Project Files (Source of Truth)

---

## 1. Назначение документа

Данный документ описывает **каноническую структуру Project Files** и назначение каждого артефакта.

Цель:
- исключить путаницу между типами документов;
- обеспечить быстрый bootstrap новых сессий и участников;
- зафиксировать, какие файлы являются нормативными, а какие — состоянием проекта.

**Важно:** Project Files являются **единственным Source of Truth**. Чаты, переписка и рабочие сессии источником истины не являются.

---

## 2. Классы документов

В проекте используются **четыре строго различимых класса документов**.

---

## 🟥 A. Governance / Meta (нормативный слой)

Назначение: правила, политики и протоколы, определяющие способ работы над проектом.

**Свойства:**
- меняются редко;
- применяются ко всем сессиям;
- имеют приоритет при конфликтах.

**Файлы:**
- `canonical_artifact_registry.md`
- `context_bootstrap_protocol.md`
- `session_closure_protocol.md`
- `session_finalization_playbook.md`
- `documentation_rules_style_guide.md`
- `versioning_policy.md`
- `meta_bundle_compressed_v1.1.md`

---

## 🟧 B. Project State (As-Is)

Назначение: фиксация **текущего состояния проекта**.

**Свойства:**
- отражают реальность «как есть»;
- обновляются по результатам сессий;
- используются для восстановления контекста.

**Файлы:**
- `project_summary_vX.Y.md`
- `project_history_log.md`

---

## 🟨 C. Project Design (To-Be)

Назначение: описание целевой архитектуры и плана реализации.

**Свойства:**
- определяют, *как должно быть*;
- меняются осознанно и версионируются;
- не обязаны совпадать с текущим состоянием.

**Файлы:**
- `pdf_extractor_techspec_v2.4.md`
- `pdf_extractor_plan_v2.3.md`

---

## 🟩 D. Infrastructure State (As-Is)

Назначение: фиксация состояния среды выполнения.

**Свойства:**
- обновляются только при реальных изменениях инфраструктуры;
- критичны для DR и восстановления;
- не содержат проектных решений.

**Файлы:**
- `server_latvia_summary_v2.2.md`

---

## 3. Правила работы с Project Files

1. **Один файл — одна роль.**  
   Документ не может одновременно быть governance и state.

2. **Имя документа является частью контракта.**  
   Неверное имя = документ некорректен.

3. **В Project Files хранится только актуальное состояние.**  
   Старые версии не должны конкурировать с текущими.

4. **Изменения фиксируются осознанно.**  
   Любое обновление сопровождается version bump и, при необходимости, Session Closure Log.

---

## 4. Минимальный bootstrap новой сессии

Для корректного начала новой сессии достаточно:

1. `project_summary_vX.Y.md`
2. `project_history_log.md`
3. `meta_bundle_compressed_v1.1.md`

При необходимости добавляются:
- TechSpec / Plan (Design);
- Server Summary (если были инфраструктурные изменения).
